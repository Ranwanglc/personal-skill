# 流程 D：RESEARCH（调研分析）—— 产出结论报告（非代码交付）

> 加载条件：PM 判定任务类型为 RESEARCH（调研/分析/技术选型/可行性评估）时加载本文件。先读主 SKILL.md 的铁律 + 共同约定，写工单时读 `references/issue-template.md`。

## 与三条开发流程的关键差异（先内化）

- **交付物是「报告/结论」，不是代码**：产出是调研报告 md（结论 + 依据 + 数据 + 建议），**不 push fork、不开 PR、不部署、不跑测试**。铁律里「PR 后置 / draft 转正式 / 部署在测试之前」这几条**在本流程不适用**。
- **仍走 multica issue + webhook + 巡检**：调研工单同样是自包含 multica issue，状态由 multica webhook 自动回报，巡检员照常用 multica-cli 揪停滞。
- **验收核心 = 结论可信 + 依据可追溯**：不是「能跑」，而是「结论有据、来源可核、覆盖问题、给出可执行建议」。
- **禁止臆造**：结论必须有来源（文档/代码/实测/数据），不确定要标「推断/待验证」，区分「事实 / 推断 / 建议」。

## 步骤

1. [octo] 老板/PM 提出调研问题 → PM 澄清**调研边界**：要回答的核心问题、范围、约束、期望产出形式（对比表/选型建议/可行性结论）、deadline。
2. [octo→multica] PM 出**自包含调研工单**（背景 + 核心问题清单 + 范围边界 + 已知约束 + 期望产出结构 + 可用信息源/仓库/文档链接 + 验收标准）。**无 push fork / 无 PR / 无部署**——工单里显式写明「交付物 = 调研报告 md，产出后随 webhook 回报，不开 PR」。
3. [multica] 调研 agent 接单 → 收集信息（读代码/文档/web/实测）→ 交叉验证来源 → 产出**调研报告 md**（结论先行、依据分层、区分事实/推断/建议、附来源）→ issue 进 in_review。**状态由 multica webhook 自动回报**（报告内容/链接随回报带回 octo）。
4. [multica] Reviewer（审核小队-review）复核报告：**结论是否有据、来源是否可核、是否覆盖核心问题、有无臆造**。有硬伤 → CHANGES_REQUESTED 打回 rerun；通过 → 下一步。
5. [octo] PM 把报告结论汇总回写群，@老板。**老板确认结论**（认可/追问/要求补充调研）。老板要求补充 → PM 建后续调研工单；认可 → done。

## 要点
- 验收核心 = **结论可信 + 依据可追溯 + 覆盖核心问题**，不是「能跑」。
- **无 PR / 无部署 / 无测试**：交付物是报告 md，随 webhook 回报，Reviewer 只审报告质量。
- 报告结构：**结论先行 → 分层依据 → 区分事实/推断/建议 → 附来源链接**（对齐老板固化的输出结构）。
- 常作为 FEAT / 方案设计的**前置**：调研结论支撑后续方案设计与开发决策；调研本身不改任何代码。
- **状态回报靠 multica 自身 webhook**，issue 到终态自动回报（报告结论随回报带回）；巡检员用 multica-cli 定时查、专治卡死。
