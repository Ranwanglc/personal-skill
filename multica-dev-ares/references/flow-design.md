# 流程 E：DESIGN（方案设计）—— 产出技术方案（非代码交付）

> 加载条件：PM 判定任务类型为 DESIGN（技术方案设计/架构设计/接口契约设计）时加载本文件。先读主 SKILL.md 的铁律 + 共同约定，写工单时读 `references/issue-template.md`。

## 与三条开发流程的关键差异（先内化）

- **交付物是「技术方案文档」，不是代码**：产出是方案设计 md（目标 + 架构/流程 + 接口契约 + 关键技术选型 + 风险与权衡 + 落地拆分建议），**不 push fork、不开 PR、不部署、不跑测试**。铁律里「PR 后置 / draft 转正式 / 部署在测试之前」在本流程**不适用**。
- **仍走 multica issue + webhook + 巡检**：方案工单是自包含 multica issue，状态由 multica webhook 自动回报，巡检员照常用 multica-cli 揪停滞。
- **验收核心 = 方案可落地 + 覆盖需求 + 风险标清**：能被后续开发直接拆成工单执行，跨端契约口径钉死无歧义。
- **必标风险与权衡**：多方案时列备选 + 各自代价，推荐一个并说明理由；非常规方案要标风险（对齐 SOUL「讨论阶段创新为本，但标注风险与权衡」）。

## 步骤

1. [octo] 老板/PM 给出设计目标 → PM 澄清**设计边界**：要解决的问题、需求/约束、已有系统现状、期望产出粒度（架构级 / 接口级 / 详设级）、deadline。若前置需要调研，先走 RESEARCH 流程。
2. [octo→multica] PM 出**自包含方案设计工单**（背景 + 设计目标 + 需求/约束清单 + 现状/相关仓库文档 + 期望方案结构 + 验收标准）。**无 push fork / 无 PR / 无部署**——工单显式写明「交付物 = 方案设计 md，产出后随 webhook 回报，不开 PR」。
3. [multica] 设计 agent 接单 → 分析需求与现状 → 产出**方案设计 md**：目标 → 总体架构/流程 → 关键接口契约（字段/口径钉死）→ 技术选型与理由 → **风险与权衡（含备选方案对比）** → 落地拆分建议（可直接转成开发工单的粒度）→ issue 进 in_review。**状态由 multica webhook 自动回报**（方案内容/链接随回报带回 octo）。
4. [multica] Reviewer（审核小队-review）复核方案：**是否覆盖全部需求、契约有无歧义、选型理由是否成立、风险是否标清、能否直接落地拆单**。有硬伤 → CHANGES_REQUESTED 打回 rerun；通过 → 下一步。
5. [octo] PM 把方案汇总回写群，@老板拍板。**跨端契约/技术选型等关键决策必须老板 approve**（认可 / 要求调整 / 选备选方案）。认可 → done；方案获批后可**直接进入 FEAT/IMPROVEMENT 流程按拆分建议派开发工单**。

## 要点
- 验收核心 = **可落地 + 覆盖需求 + 契约无歧义 + 风险标清**。
- **无 PR / 无部署 / 无测试**：交付物是方案 md，随 webhook 回报，Reviewer 只审方案质量。
- 方案结构：**目标 → 架构/流程 → 接口契约 → 选型理由 → 风险与权衡（备选对比）→ 落地拆分建议**。
- **关键决策等老板拍板**：跨端契约、技术选型不自行钉死，写清备选与推荐由老板 approve（对齐「merge/关键决策等老板拍板」的铁律精神）。
- 常是 FEAT 的**前置**：方案获批后按「落地拆分建议」转 FEAT/IMPROVEMENT 开发工单；方案本身不改代码。
- **状态回报靠 multica 自身 webhook**，issue 到终态自动回报（方案随回报带回）；巡检员用 multica-cli 定时查、专治卡死。
